Days 1-30

Read the work

Establish a truthful picture of demand, workflow friction, ownership, architecture, governance, and existing capability.

Priority actions

  • Map active AI initiatives, pilots, vendors, owners, project/function sponsors, and current disposition.
  • Shadow representative workflows across engineering, project controls, construction, information management, quality, and handover.
  • Align with Data Architecture & AI on approved platforms, integration patterns, security, evaluation, support, and escalation boundaries.
  • Define a lightweight intake record and work-package completeness standard.
  • Select two bounded candidates with named workflow owners and measurable consequences.

Outputs

  • Current-state AI portfolio map
  • Stakeholder and decision-rights map
  • Work-package intake standard
  • Two discovery briefs with baseline evidence

Discovery questions

  • Which existing initiatives already have reusable components?
  • Where is duplicate demand hidden by different functional language?
  • Which controls are non-negotiable, and where is local discretion expected?

Operating posture

Listen before standardizing. Treat existing practices as adaptations to real project constraints until evidence proves otherwise.

Success evidence for this phase

Stakeholders recognize the portfolio view as accurate; selected workflows have named owners, consequences, and baselines.

Days 31-60

Prove the package

Deliver controlled real-work tests that produce evidence, learning, and ownership—not showcase demos.

Priority actions

  • Run discovery workshops at the workflow artifact and decision level.
  • Choose the smallest useful intervention: guidance, copilot, agent, low-code app, automation, lightweight model, or no build.
  • Define human authority, fallback, source provenance, evaluation, support, and change-control expectations before release.
  • Instrument value categories and correction signals in the workflow cadence.
  • Capture reusable components and lessons as the build progresses.

Outputs

  • Two controlled workflow interventions
  • Evaluation and human-authority records
  • Value baseline and evidence plan
  • Initial reusable pattern library

Discovery questions

  • What would cause the project owner to stop using the solution?
  • Which failure must trigger fallback or escalation?
  • What can be standardized without erasing project context?

Operating posture

Build with workflow owners, not for them. Keep authority, evidence, support, and correction paths visible.

Success evidence for this phase

The interventions are used in real work; users can correct them; evidence is available for an explicit disposition.

Days 61-90

Standardize the learning

Turn evidence into a durable portfolio cadence, reusable capability, and practical adoption system.

Priority actions

  • Review results with project owners, Digital Solutions, Data Architecture & AI, and affected functions.
  • Disposition each initiative: advance, revise, reuse, hold, stop, or transfer to another accountable team.
  • Package successful components with ownership, training, support, evaluation, and value receipts.
  • Launch portfolio review cadence and transparent value-realization reporting.
  • Recommend a 12-month sequencing logic grounded in demand clusters, reuse, consequence, readiness, and adoption burden.

Outputs

  • First reusable capability work package
  • Portfolio review and value cadence
  • Guidance and training starter set
  • 12-month sequencing recommendation

Discovery questions

  • What evidence earns wider deployment?
  • Who owns the capability after the enablement team steps away?
  • How will lessons move across projects and GBUs without becoming bureaucracy?

Operating posture

Scale proven patterns, not enthusiasm. Preserve the reason a local variation exists before deciding whether to standardize it.

Success evidence for this phase

At least one capability is reusable with clear ownership; the portfolio cadence closes decisions and makes value and learning visible.
